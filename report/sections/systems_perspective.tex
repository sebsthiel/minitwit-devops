\section{System's Perspective}
\subsection{Design and Architecture}
\label{sec:design}

The Minitwit system is deployed on DigitalOcean using three droplets: one PostgreSQL database node, one Docker Swarm manager node, and one Docker Swarm worker node. The application is containerized and deployed through Docker Swarm, which runs and manages the application services across the swarm.

The original plan was to use Vagrant with DigitalOcean, but this caused compatibility issues, especially on Windows machines. Instead, we implemented a PowerShell-based deployment script, \texttt{scripts/deploy-existing.ps1}. The script deploys to already existing droplets by connecting to the server with SSH, copying configuration files with SCP, setting the required environment variables, and deploying the services using Docker Swarm.

The public entry point to the system is Nginx. Incoming traffic is routed through Nginx, which forwards requests 
to the web service, the API service, or Grafana based on the URL path. The API and web services communicate with the PostgreSQL database for persistent storage. The simplified request flow is:

\begin{center}
    \texttt{Browser / Simulator $\rightarrow$ Nginx $\rightarrow$ Web / API $\rightarrow$ PostgreSQL}
\end{center}

The system also includes a full monitoring stack comprising Prometheus, 
Grafana, Loki, Promtail, cAdvisor, and node-exporter, which is described 
in detail in Section~\ref{sec:monitoring}.

The deployment architecture is illustrated in Figure~\ref{fig:deployment-diagram}.

A \texttt{scripts/create-infrastructure.ps1} script was also implemented to automate full infrastructure creation from scratch. It creates the three droplets, installs PostgreSQL and Docker, configures Docker Swarm, joins the worker node, and restores the database from a backup. A companion script \texttt{scripts/backup-database.ps1} backs up the live PostgreSQL database before any redeployment. Testing of the full infrastructure creation was limited because the DigitalOcean account reached the three-droplet limit, preventing creation of new droplets while the existing ones were running.

\subsection{Dependencies}

The following are the dependencies for our Minitwit system, listed in an organized manner in order to better provide an overview.

\textbf{Runtime Dependencies}

These dependencies include the environment to run the application on including the operating system, VM technologies and programming language.

\begin{itemize}
    \item Ubuntu Linux: Operating System
    \item Docker (swarm mode): Virtual Machine
    \item golang:1.25 (Docker image)
\end{itemize}

\textbf{Application Libraries}

This list contains the libraries and what each library is for.

\begin{itemize}
    \item Go 1.25.6: Programming language.
    \item Gorilla Mux: HTTP request routing.
    \item Gorilla Sessions: Session management.
    \item GORM: Object-relational mapper.
    \item PostgreSQL Driver: Database connectivity.
    \item Zerolog: Structured logging.
    \item Prometheus client\_golang: Metrics instrumentation.
\end{itemize}

\textbf{Testing}

\begin{itemize}
    \item Python 3.11: Programming language
    \item pytest: Testing framework
    
\end{itemize}

\textbf{DevOps / Build Dependencies}

This list includes tools for employing DevOps practices. 

\begin{itemize}
    \item Github: Code repository
    \item Dockerhub: Docker image repository
    \item Github Actions: CI/CD Pipeline
    \item Codacy: Quality assurance
    \item SonarQube: Quality assurance
    \item Trivy: Pipeline hardening
    \item Fail2Ban: Intrusion-prevention framework
\end{itemize}

\subsection{Current System State}

The application has certain issues, which have not been resolved due to lack of time. These issues are mentioned below.

\begin{itemize}
    \item Ideally we would have more nodes, and more managers to ensure that even if a manager dies, another manager can still remain in control of the swarm.
    \item There is a bug where it is possible to create multiple instances of the same user.
    \item There are currently no logging for the WEB part of the application. This has been neglected as almost all traffic has come from the simulator, which only talks with the API.
    \item Our entire monitoring stack as well as nginx don't have a health check configured, meaning we cannot monitor the uptime of these services
    \item We have used Trivy for analysing dependencies in in the docker images, but have not acted on the information.
\end{itemize}
